# PM-018 — Resurrection Mechanisms Cascade + Duplicate Liquidation Race

*(PM-017 was reserved for the initial watchdog incident; same causal chain, merged here.)*

**Incident window:** 2026-07-07 → 2026-07-10
**Environment:** Paper trading account
**Financial impact:** ≈ 0 (paper) — NAV unchanged within noise
**Epistemic impact:** 3 days of non-deterministic broker state

---

## Summary

An operator-initiated halt (`systemctl stop` + `disable`) failed to keep the
trading service down. Investigation revealed **four independent, uncoordinated
resurrection mechanisms**, each capable of restarting the service. Separately,
a liquidation script launched twice re-submitted orders against already-flat
positions, inverting the book into unintended shorts. The fault class had been
specified days earlier as FAULT-03 (*concurrent instance race*) in the
fault-injection harness — production reproduced it through three distinct vectors.

## Resurrection mechanisms identified

| # | Mechanism | Why `disable` didn't stop it |
|---|-----------|------------------------------|
| M1 | Cron watchdog (`*/5 min`) | Issues explicit `systemctl start` — `disable` only blocks boot-time activation |
| M2 | `Restart=always` / `on-failure` (unit + drop-in) | Auto-restarts after any non-zero exit |
| M3 | Supervisor unit (`Wants=` + internal start loop) | Pulls the target unit up every ~30s |
| M4 | Control API with start/restart endpoint | Out-of-band explicit starts |

## Duplicate liquidation race

Two instances of the same liquidation script ran concurrently (launched
2 minutes apart). Overnight orders had already filled at market open; both
instances re-read stale position state and re-submitted identical orders
against now-flat positions — inverting long exposure into shorts. A lock
guard from a prior incident existed, but protected a *launcher script*,
not the shared resource. The new entry path bypassed it entirely.

## Root cause

Protection was attached to **entry points** (launchers, service definitions)
instead of the **shared resource** (order ledger / broker session).
Entry points multiply; the resource does not.

## Remediation

1. All four resurrection mechanisms eliminated.
2. Critical units **masked** (`systemctl mask` → `/dev/null`): any restart
   now requires a deliberate, auditable `unmask`.
3. Operator-halt flag checked every cycle.
4. Single-writer enforcement moved to resource level (`flock` on the ledger).
5. Side-effectful broker scripts must re-read broker state **at action time**,
   not at launch time.
6. Full manual liquidation executed interactively: all fills confirmed,
   broker verified flat (zero positions, zero orders).

## Lessons

- `disable` ≠ prohibition. **`mask` = prohibition.** A kill-switch is only
  real if the complete start-path graph is controlled.
- Single-writer guarantees belong on the resource, never the launcher.
- Idempotence requires reading the world at action time, not at plan time.
- A specified-but-not-enforced fault class **will** occur in production.
  Test harnesses predict; architecture prevents.

gate remain blockers before any restart.
